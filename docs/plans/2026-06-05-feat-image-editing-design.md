# feat: Add image-to-image and lightweight PS edits

## Summary

Extend Synthetic Vision from pure text-to-image into a staged image editing workspace:

1. **Image-to-image / reference remix**: upload a source image, describe the desired change, and generate a new image from that reference.
2. **Lightweight PS edits**: draw a mask on top of the source image, describe the local replacement/removal, and send the source + mask through the same async generation pipeline.

This keeps the existing SaaS shape intact: authenticated Dashboard flow, credit deduction/refund, async worker queue, Gallery results, and pluggable mock/OpenAI providers.

## External API facts used

OpenAI's Image API exposes separate generation and edit paths. Generations create images from prompts; Edits modify existing images with a new prompt. The official guide recommends the Image API for a single image generation/edit from one prompt, while Responses API is better for conversational multi-turn editing.

For image references, the Image API uses `/v1/images/edits` with multipart `image[]` inputs and `prompt`. For local PS edits, `/v1/images/edits` also accepts `mask`; the mask applies to the first input image, is prompt-guidance rather than pixel-perfect control, and must match the edited image's format/size while containing an alpha channel.

References:

- https://developers.openai.com/api/docs/guides/image-generation#overview
- https://developers.openai.com/api/docs/guides/image-generation#create-a-new-image-using-image-references
- https://developers.openai.com/api/docs/guides/image-generation#edit-an-image-using-a-mask

## Current repo context

Current implementation is text-to-image only:

- `backend/internal/provider/provider.go` exposes `Provider.Generate(ctx, GenerateRequest)` with prompt/style/size/seed only.
- `backend/internal/provider/openai.go` posts JSON to `{OPENAI_BASE_URL}/v1/images/generations` only.
- `backend/internal/service/service.go` stores one generated output per `Generation` and has no source/mask metadata.
- `backend/internal/handlers/generation.go` accepts JSON only for `POST /api/generations`.
- `frontend/src/views/Dashboard.vue` already has an "添加参考图片" visual button, but it is not wired to file upload.

## Goals

- Add useful **图生图** without disrupting existing 文生图.
- Add a first-pass **PS mask** workflow that feels native to the Aurora Grid/Cyber Neon UI.
- Preserve async generation, credit deduction, refunds, and Gallery result handling.
- Keep uploads bounded and private enough for v1: source/mask files are stored under `DATA_DIR/references`, not exposed as public static assets by default.
- Make the mock provider support the new modes so the app remains fully usable without an external API key.

## Non-goals

- No full Photoshop clone: no layers, history stack, crop/outpaint toolset, filters, or asset library.
- No multi-reference support in the first implementation. The backend shape should not block it, but UI starts with one source image.
- No new billing model for edits. Cost remains resolution-based for v1.
- No public source-image serving endpoint in v1. The generated output remains public under `/images/<id>.png`, matching current behavior.
- No migration away from the existing Image API provider in this slice.

## Selected approach

Use one staged contract around the existing `POST /api/generations` endpoint.

### A. Text mode remains JSON-compatible

Existing clients keep sending:

```json
{ "prompt": "...", "negative_prompt": "...", "style": "Cinematic", "resolution": "2K", "aspect_ratio": "1:1" }
```

The backend treats missing `mode` as `text`.

### B. Image/edit modes use multipart form-data

For source-image workflows, the frontend sends multipart form data to the same endpoint:

| Field | Required | Notes |
|---|---:|---|
| `mode` | yes | `image` for reference remix, `edit` for local PS mask |
| `prompt` | yes | Edit/remix instruction |
| `negative_prompt` | no | Same meaning as text mode |
| `style` | no | Existing style label |
| `resolution` | yes | Existing `1K/2K/4K` |
| `aspect_ratio` | yes | Existing `1:1/4:3/16:9/9:16` |
| `source_image` | yes for image/edit | PNG/JPEG/WebP, bounded by size limit |
| `mask_image` | yes for edit | PNG with alpha channel, generated by the frontend canvas |

### C. Generation metadata grows minimally

Add these fields to `Generation`:

- `Mode string` JSON `mode`: `text`, `image`, `edit`.
- `SourceImagePath string` JSON omitted: server-private stored upload path.
- `MaskImagePath string` JSON omitted: server-private mask path.
- `HasSourceImage bool` JSON `has_source_image`.
- `HasMaskImage bool` JSON `has_mask_image`.

This gives Gallery/Analytics enough mode metadata without leaking uploaded sources.

### D. Provider request carries optional images

Extend `provider.GenerateRequest` rather than replacing the interface:

- `Mode string`
- `SourceImage *provider.InputImage`
- `MaskImage *provider.InputImage`

`OpenAIProvider.Generate` branches:

- no `SourceImage`: current `/v1/images/generations` JSON request.
- source/mask present: `/v1/images/edits` multipart request with `model`, `prompt`, `image[]`, optional `mask`, optional size/output format fields when accepted by the configured model.

`MockProvider.Generate` remains deterministic but incorporates mode/source/mask signal visually so tests and demos show distinct behavior.

## UX design

### Dashboard modes

Add a compact segmented control above the prompt canvas:

- `文生图`: current behavior.
- `图生图`: enables source upload and reference remix presets.
- `局部修图`: enables source upload plus mask drawing.

### Source upload

- Reuse the existing "添加参考图片" button as the primary upload entry.
- Accept image drag/drop and file picker for `image/png`, `image/jpeg`, `image/webp`.
- Show a preview card with filename, approximate dimensions if readable, and a remove action.
- In `图生图` / `局部修图`, generation is disabled until a source image exists.

### Lightweight PS mask

- In `局部修图`, render the source image in the canvas region with a translucent cyan/pink mask overlay.
- Provide brush size, erase/paint toggle, clear mask, and quick action chips:
  - `替换选区`
  - `移除物体`
  - `扩展背景`
  - `重绘材质`
- Export the mask as a PNG with alpha. The transparent/alpha area is the edit guide sent to the provider.

### Error states

- Unsupported file type: show inline error before request.
- Oversized upload: show inline error before request.
- Missing source/mask in image/edit modes: disable submit and show helper copy.
- Provider user error: keep current refund behavior and show the provider error on the generation row.

## Backend design

### Upload validation

- Max upload size: 10 MB per source/mask in v1.
- Accepted source MIME: `image/png`, `image/jpeg`, `image/webp`.
- Accepted mask MIME: `image/png` only.
- Read a bounded in-memory copy in handler/service; write persisted files under `DATA_DIR/references` using generation ID based filenames.
- Avoid trusting user filenames except for extension inference.

### Service flow

1. Validate mode and core generation params.
2. Validate required source/mask presence by mode.
3. Compute cost/dimensions/seed exactly as today.
4. In one DB transaction:
   - load user and deduct credits,
   - insert `CreditTransaction(reason="generation")`,
   - insert pending `Generation` with mode and source/mask flags,
   - persist source/mask files under `DATA_DIR/references`,
   - update private path fields.
5. Enqueue the generation after transaction success.
6. Worker loads private source/mask files when present and passes them to provider.
7. Existing success/failure/refund behavior remains unchanged.

### Provider behavior

- Existing JSON generation path stays intact for text mode.
- Edit path uses multipart form data and sends `image[]` for source image.
- If a mask exists, send `mask` along with the first image.
- Decode `data[0].b64_json` first; retain current URL fallback/download guard if a compatible gateway returns URL output.

## Frontend/API design

### Types

Add:

```ts
export type GenerationMode = 'text' | 'image' | 'edit'
export interface Generation { mode:GenerationMode; has_source_image:boolean; has_mask_image:boolean; ... }
export interface CreateGenInput { mode?:GenerationMode; source_image?:File; mask_image?:Blob | File; ... }
```

### API client

`GenAPI.create` decides request shape:

- no source/mask: current JSON `POST /generations`.
- source/mask: multipart `FormData` POST `/generations`.

### Store

`generations.create(input)` remains the single public store action so Dashboard does not branch around API details.
